Java日志系统

java 日志系统

log4j

基本理念，构成：

  • Logger：记录日志的对象

  • Appender：日志输出源

  • Level：日志级别

jul - java.util.logging

官方日志系统，借鉴log4j的理念，使用上也大同小异

jcl - Jakarta Commons Logging

为了解决jul出现之后，带来的java日志系统的混乱。Apache推出commons-logging框架，来解决连接和适配的作用。

类似一个公共的接口，而log4j和jul都是具体的实现。

LogFactory会在运行期间自动查找具体的日志实现，按照一个设定的路径进行匹配：

  1. 查找 org.apache.commons.logging.Log 配置项，根据配置项的值来直接获取具体的日志实现。配置项可以在代码中，或者在 commons-logging.properties 配置文件中；

  2. 在系统变量中查找 org.apache.commons.logging.Log 配置项；

  3. 如果当前项目下使用的是 log4j 日志系统，那么使用 log4j 作为实现类，这里使用的是包装类 Log4JLogger；

  4. 如果当前项目下使用的是 jul 系统，那么使用 jul 作为实现类，这里使用的是包装类 Jdk14Logger；

  5. 如果以上全部不匹配，那么使用最简单最基础的日志类 SimpleLog。

上面的 Log4JLogger、Jdk14Logger 与 SimpleLog 都位于 org.apache.commons.logging.impl 包下，是 jcl 中自带的包装类，它们都实现了 org.apache.commons.logging.Log 接口，并分别包装了对应的日志对象，最后将日志操作转交到日志对象上进行。例如 Log4JLogger 中包装了 log4j 的日志类，最终的日志操作是由 log4j 来完成。

这也正是 jcl 的简单原理，它抽象出一个通用的日志接口，对每个日志的具体实现进行了适配，开发者统一使用接口层，并且可以自由配置具体的日志实现。

slf4j 与 logback

jcl 貌似已经解决了问题，但事实并非如此。因为jcl是在运行期间查找符合条件的日志类，然后使用类加载的方式加载指定类。但是类加载会带来一些不可预知的错误，而且增加了调试难度。具体细节日后补充。

所以log4j的作者发起来slf4j的项目。同样是提供一个公共的接口，开发者只需要关注上层的接口，无需考虑下层的实现。只要slf4j的实现遵循一个接口，各个不同的日志系统之间可以无缝连接。

logback 就是 slf4j 的官方实现，对比 log4j 有更强的性能。==slf4j + logback== 也正在成为最流行的日志框架选择。

slf4j 和 jcl 的实现原理到底有什么不同呢？jcl 是通过类加载的机制，在运行期间动态查找到具体的实现类。而 slf4j 是在==编译期间对具体的实现类进行绑定==。编译时，它会查找类路径下的 org.slf4j.impl.StaticLoggerBinder，绑定工作都会在这个类中进行。如果类路径下有多个 StaticLoggerBinder，编译器会给出警告。因此 sjf4j 是在编译期间静态绑定具体的 Log 实现，从而无需在运行期间进行动态查找，这样不仅避免了类加载过程中的各种问题，也提升了性能。

log4j2

log4j2 又是 log4j、slf4j、logback 的作者的新作品。它对 log4j 做了很大的改进，提供了多种现代特性，而且它的异步性能优越，在分布式系统中性能要远好于 logback，同时也支持 slf4j 与 jcl。

总结

  • log4j 打开日志系统的大门，成为后面的标准。

  • jul 的官方实现，与log4j 不兼容，造成了日志的混乱。

  • jcl 提供抽象接口来统一日志系统。

  • jcl 基于运行期间类加载的机制，会有问题，不够好，所以 log4j 的作者写了一个抽象框架 slf4j 并直接提供了一套实现，也是新的日志系统，logback。

  • 后来作者改进了 log4j，推出log4j2。

另外，前面提到了日志系统和日志框架两个名词，这是两个不同的东西：

  • 日志框架：只是一种日志接口，不负责具体的日志输出实现，而是通过配置来灵活的选择具体的日志系统。例如 jcl、slf4j 就是日志框架，它们只提供了接口，没有提供具体的日志系统实现；
  • 日志系统：真正的日志实现，如 log4j、jul、logback 与 log4j2 都是具体的日志系统实现。

日志框架与系统，可以理解为 Java 中的接口与具体实现类。理想的日志使用模式是，代码中全部使用日志框架的 API 接口，在配置文件中配置具体的日志系统实现，这样可以通过配置来灵活切换不同的日志系统，而不用修改代码。

另外我们知道日志框架与日志系统是相互配合来使用，这其中会通过适配与桥接来完成。以 slf4j 举例来说：

  • 适配就是选择 slf4j 具体的日志实现类，例如使用 logback-classic 会选择 logback 作为日志实现，而使用 slf4j-log4j12 会选择 log4j 作为日志实现。
  • 桥接是将当前其他的日志框架或者日志系统，最终转发到 slf4j 上。例如原系统使用 jcl，那么可以通过 jcl-over-slf4j 桥接到 slf4j 上，最终日志请求会从 jcl 转发到 slf4j 上，再适配到具体的日志实现上，最终得到日志输出。

slf4j + logback

这是一种常用的日志使用方式，在较新的应用中越来越流行。它使用 slf4j 的日志 API 接口，使用 logback 作为具体的日志实现。该方式需要的包也非常简单：

  • slf4j-api
  • logback-core
  • logback-classic

一个具体的日志操作的过程，大概是首先调用 slf4j 的 API 接口，然后 slf4j 将请求转发给 slf4j-log4j12 来处理，这个是在编译期间静态绑定好的。然后 slf4j-log4j12 最后通过 log4j 来完成日志操作。

Https_frontend_encryption 前端加密还是后端加密

HTTPS，前端是否需要加密

分析在HTTPS的环境下，对于用户登录这类，涉及隐私信息(密码)的场景，我们是否有必要进行前端加密。

首先这个加密是宽泛意义的，不是特指安全，密码学的相关知识，还包括简单的不可逆的hash。

其重点是将明文密码隐秘化。

什么是HTTPS

在分析之前，先简单介绍一下HTTPS。在之前使用不加密的HTTP通信，就是信息的明文传播，带来了三大风险：

  1. 窃听风险
  2. 篡改风险
  3. 冒充风险

HTTP+SSL=HTTPS，为了解决这三大问题，希望达到：

  1. 信息是加密的，无法被窃听
  2. 具有校验机制，被中间人篡改会被发现
  3. 配备身份证书，防止中间人冒充

细节部分可以自行查阅或者查看参考链接，简单来说就是非对称加密+对称加密。

为了确保非对称加密中，双方的公钥在传输过程中不被劫持篡改，把公钥放在数字签名中，只要证书是可信的，那么公钥就是可信的。

根据第一次客户端到服务器的随机数，第二次服务器到客户端的随机数，加上第三次客户端生成的随机数(“pre-master key”)，客户端生成会话的对称密钥，通过服务器的公钥加密，然后发送给服务器，服务器收到后，用私钥进行解密，获取到对称密钥。

到了这一步之后，就可以利用对称密钥进行加密通信，可以完全使用普通的HTTP协议进行传输。

信道安全，前端要不要加密

本人一开始的看法是，不需要加密。

因为在HTTPS的保障下信道安全的角度下，传明文还是一个”加密”的密码没有太大的意义，就算在服务器端出现漏洞，泄露了密码，加密的密码其实也可以看成就是你设置的一个随机的密码。原来我需要知道你们的密码明文来登录，现在我用加密后的密码来登录，其实并没有什么区别。

对于整个系统来说，安全性并没有因为上传密码之前做一次加密而提高。

而一般来说，我们需要考虑的是数据库被托库之后，保证用户的密码不被窃取。所以主流的做法是：

  1. 数据库切不可存明文
  2. 可以多次hash(MD5)提高攻击成本
  3. 可以针对每一个密码进行加salt处理

好的，后端怎么做没有什么疑惑了，那么前端到底要不要加密，是不是真如我所想，不需要加密呢？

这里我在知乎(币乎)搜了一下，有一个相似程度极高的问题。所以我看了一下，并做了思考：

WEB前端密码加密是否有意义？

既然前端的密码都是直接可以看到的，那么加密是否还有意义

我浏览了一下答案后，很多答案都提到了eldereal的答案，所以我找到他的答案：

这个回答是2014年，他的角度主要是不能依赖前端加密而后端什么都不做，也就是把后端加密的工作挪到前端是含无意义的。其次是在http明文传输的协议下，前端的简单加密或者说hash是没有作用，并不能保障系统的安全。

最后这段话，我是很赞同的。

由于前端是不是加密，所有安全机制都必须照常应用，所以为系统增加这样的复杂性是完全没必要的，即使传输明文密码，只要正确使用了HTTPS连接和服务器端安全的哈希算法，密码系统都可以是很安全的。

好，这是不要加密的阵营，我还发现知乎大V轮子哥的发言：

时间线离我们很近，还是相同的理由。

下面我搜寻要加密的阵营的答案：

答主是一个安全相关专业的学生，他的观点指出前端的加密不是简单的hash(MD5)那么简单就可以概括的，比如说可以收到服务器传递的随机数salt再hash保证窃听到的hash值每一次都是不一样的。

同时他也指出在https的大环境环境下，前端加密与否更重要的在于保证服务器数据的隐秘性。在HTTPS保证信道安全的基础上，不希望网站可以知道用户的明文密码。

所以我认为有了HTTPS，已经能够防止中间人的窃听，这首先不需要前端来加密实现，而且仅仅靠前端的加密(即使依赖相关密码学知识)也是无法抵御中间人问题，究其所以是因为信任的问题。正也是为什么HTTPS需要数字证书来保证信任。所以仅剩下的前端加密的真正价值可能只是一种对用户来说的隐私保护服务。

总结

综上所述，我认为在HTTPS的环境下，前端加密是存在意义的，这个意义不在于是来保证网站的一个安全性，更多的是提供一种对用户来说的隐私保护服务。

至于要不要前端加密，就取决于当前具体的业务。如果用户有想要保护隐私的一个强烈的诉求，可以。但是如果没有，也可以不加密，舍弃这些相关的逻辑，也不会对网站的安全造成较大的影响。
